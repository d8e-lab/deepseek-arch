# 粘贴重构回顾：plan.skill 与协作反思

> v1.2.0 · 2026-06-16

## plan.skill 评估

### 做得好的

**复杂度判定准确**。粘贴重构涉及 4 个功能点（两个 bug + 直接粘贴 + #N 标记 + 原子删除），正确判为"复杂"进入全流程，避免了反面案例中"跳过规划直接改"的错误。

**Phase 2.5 自检修正确保了范围不膨胀**。原始需求隐含了"跨平台适配"，自检时明确"不做系统信息检测，改用输入归一化"，省去引入新模块的工程开销。

**确认断点机制有效**。Phase 2.5 输出计划后等待用户确认再执行，避免了方向性返工。对比 `/model` 命令的教训（自行脑补弹窗 UI 多写 100+ 行），这次没有偏离。

**子任务拆分粒度合理**。5 个 commit 各自独立可编译可测试，没有出现"连写 3 个 commit 然后修补"的情况。

### 可以改进的

**1. 缺少"挑战用户技术假设"的步骤**

用户提出"是不是还要先抓取系统信息"。plan.skill 的 Phase 0 有"识别模糊点，主动追问"，但没有"识别用户提出的技术方案是否过度"这一步。实际执行中我在 Phase 1 自己判断了不做系统检测，但 skill 本身没有把这个判断流程化。

**建议**：Phase 1 增加一条检查——"用户提出的技术方案是否是最简实现？如果不是，给出更简方案及理由"。

**2. 计划输出篇幅偏大**

对于一个最终只改了 1 个核心文件（+110 行）的任务，Phase 0-2.5 的完整输出约 80 行。用户需要读完才能说"开始"。对于了解代码库的开发者，这有 overhead。

**建议**：在确认断点前加一句 TL;DR——"计划总结：改 2 个文件，5 个 commit，预计 +150 行。详细计划如下。需要调整请说，否则可以直接说'开始'。"

**3."复杂"判定门槛可以更细粒度**

当前判定：功能点 ≥ 2 即复杂，进入全流程。但"修两个小 bug + 加一个小功能"和"重构核心架构"的复杂度完全不同。可以用改动量辅助判定（如预计 < 100 行则降级为"中等"，精简流程）。

**建议**：增加一条降级规则——"如果判定为复杂但预计改动 < 3 文件且 < 100 行，可降级为中等流程：跳过 Phase 2.5 自检表，但仍保留确认断点。"

**4. Phase 3 执行部分缺乏增量指导**

Phase 3 说的"每步编译测试提交"是常识。粘贴重构中实际的执行挑战——比如原子删除的 `findPasteMarkerAt` 要处理 `'left'` 和 `'right'` 两种语义、跨行场景的判断——这些细节 skill 没有帮助预判。

**建议**：Phase 2 子任务表中"风险与缓解"列应该更具体，不是泛泛的"注意边界情况"，而是"Backspace 在标记末尾时 idx 的边界条件需要单测覆盖"。

## 用户指挥评估

以下基于"对技术路线和细节不了解"的前提来分析。

### 做得好的

**分阶段推进**。"先完成 1 和 2，git 提交后再完成 3"——这个指令避免了原子删除和基础修复混在一起导致的定位困难。分阶段 + 中间 commit 是成熟的做法。

**追问边界条件**。"是不是还要先抓取系统信息"——虽然最终没走这个方向，但主动思考跨平台适配说明安全意识好。很多 bug 就是跨平台没考虑导致的。

**推动文档化**。"写两个文档，然后反思 plan.skill"——这是从单次任务中提取可复用经验的动作。没有这个追问，plan.skill 的改进点不会进入讨论。

### 可以改进的

**1. 技术方案用提问代替建议**

"是不是还要先抓取系统信息"这是一个带倾向性的提问——暗示"应该做系统检测"。如果执行者不够独立，可能直接照做，导致过度设计。

**更好的说法**："跨平台（Linux/Windows/WSL）的换行行为不同，你打算怎么处理？"

把"怎么做"留给执行者判断，你只提供"必须考虑什么"。

**2. 范围可以在需求阶段更明确**

原始需求是"重构粘贴功能，你觉得有没有什么问题"——这是一个开放式探索指令。然后逐步追加"多行粘贴""原子删除"等。如果一开始就说清楚"我要修 bug + 小粘贴直显 + 原子删除 + 版本号升级"，plan 会更精准。

**更好的说法**："粘贴功能有几个问题我想修：粘贴多行显示错乱、行数不准、小粘贴不直观、标记不能一键删除。你帮我分析现有代码再出方案。"

**3. 验收标准可以更具体**

"用户粘贴多行内容时会不会有问题"——这个追问很好，但后续没有给出"什么算没问题"的具体标准。最终是靠测试用例补上了验收标准。

**更好的说法**："修完之后，我期望的行为是：WSL 下粘贴 3 行 Python 代码，输入框直接显示这 3 行，提交后对话历史里正确换行。你用测试覆盖这些场景。"
