# Plan Skill — 编码任务规划与自检框架

> 每次接到编码任务时自动激活。
> 目标：用最小改动量一次做对，避免改错模块、过度设计、反复返工。

---

## 0. 复杂度快速评估（必须首先执行）

在进入任何阶段之前，判断任务复杂度。回答以下问题：

1. 该需求涉及几个独立的功能点/模块？
2. 需要修改 ≥ 3 个文件，或改变核心数据模型/接口签名？
3. 是否存在不确定因素（如未指定的技术方案、遗留代码行为未知）？

**判定规则**（满足任一即“复杂”，进入 Phase 0–4 全流程；否则“简单”，直接给出 1 句话修改方案并执行，但仍须遵守最小改动原则）：
- 功能点 ≥ 2
- 文件数 ≥ 3 或核心接口变更
- 问题 3 为“是”
- 用户原话含“设计”“架构”“重构”“多步”“方案”等词

---

## Phase 0 — Comprehend（理解需求）

```
□ 用自然语言复述需求，逐条与需求方确认
□ 明确“不要什么”和“要什么”同等重要
□ 识别模糊点，主动追问，不等需求方自己发现
□ 确认验收标准：
    - 需求方如何判断“做完了”（人视角）
    - 我如何用代码/测试验证“做完了”（技术视角，尽量自动化）
```

**强制产物**：需求确认摘要 + 人与技术双视角验收标准

---

## Phase 1 — Orient（定位）

```
□ 从 CLI/入口函数开始追踪调用链，确认运行时实际类型/实例
□ 画出数据流：输入从哪来 → 经过哪些函数 → 最终到哪
□ 标记拦截点：在调用链的哪个环节插入新逻辑
□ 如果涉及已有接口/类，确认所有实现方和调用方
```

**强制产物**：调用链草图 + 插入点标注

---

## Phase 2 — Decide & Plan（决策与拆解）

### 2.1 列出潜在受影响文件
```
□ 列出所有可能需要修改的文件
□ 对每个文件问三个问题：
    1. 不改这里行不行？
    2. 改了会不会影响其他路径？
    3. 有没有更简单的方式？
□ 削减：改动量 > 3 文件或 > 50 行时，重新审视是否过度设计
□ 生成最终文件清单，每个文件附一句话理由
```

### 2.2 拆解为子任务（复杂任务强制执行）
每个子任务必须包含以下字段：

| 字段 | 说明 |
|------|------|
| ID | 序号 |
| 目标 | 本步骤要达到的可验证结果 |
| 输入 | 依赖的前置产出物（文件、数据、接口） |
| 输出 | 完成后产生的具体产物（文件/测试/配置） |
| 验收标准 | 可自动或人工验证的条件（如“测试 X 通过”） |
| 复杂度 | 低/中/高 + 1 句理由 |
| 风险与缓解 | 本步骤特有风险及应对 |
| 顺序 | 可并行/依赖步骤 ID |

**强制产物**：文件清单 + 子任务分解表

---

## Phase 2.5 — 计划自检与完善（关键闭环）

在输出最终计划前，你必须将自己的计划与以下“理想计划原则”逐条对比，生成差异分析，并修订计划。**修订处在最终计划中用 `[→ 已优化]` 标注。**

### 理想计划原则
1. 总目标是否被拆分为可独立验证的子任务？
2. 每个子任务的验收标准是否可自动化/客观判断？
3. 是否存在隐藏依赖未显式标注？
4. 高风险步骤是否给出了替代方案？
5. 计划是否是最小可行路径（无过度设计、无顺手重构）？

### 必须输出的自检表格（复杂任务时）
| 原则 | 符合？ | 缺失点 | 优化措施 |
|------|--------|--------|----------|
| 1 可独立验证 | 是/否 | ... | ... |
| 2 自动化验收 | 是/否 | ... | ... |
| 3 依赖显式 | 是/否 | ... | ... |
| 4 高风险备选 | 是/否 | ... | ... |
| 5 最小可行 | 是/否 | ... | ... |

**基于自检表修订计划，然后输出最终版本。**

## Phase 3 — Act（执行）

### 前置强制步骤（不可跳过，必须先完成再改代码）

```
□ 1. git stash                          # 暂存未提交的修改
□ 2. git checkout -b <分支名>          # 从当前分支创建新分支（feature/* 或 fix/*）
□ 以上两步完成后，才能开始修改文件。分支是你的安全网，坏了就 git checkout -- .
```

### 执行循环

```
□ 按子任务依赖顺序执行，切勿并行违反依赖
□ 每一步循环：
    1. 做一处逻辑完整的改动
    2. 编译/构建通过
    3. 运行相关测试
    4. git add <具体文件> && git commit -m "<中文：改了什么 + 为什么>"
□ 每步检查是否偏离 Phase 2.5 的最终计划
□ 遇到失败先停：是改错了还是方案有误？
□ 不改与任务无关的文件（绝对禁止顺手重构）
```

**强制产物**：每个 commit 独立可编译可测试

---

## Phase 4 — Verify（验证）

```
□ 从入口模拟调用路径，确认新代码可达
□ grep 确认新方法调用方存在且签名正确
□ 运行全量测试
□ 对照 Phase 0 的双视角验收标准，确认全部通过
□ 自问：如果这段代码没生效，第一个排查点在哪？
□ 用自然语言向需求方描述修复后的行为
```

**强制产物**：验收记录 + 修复后数据流描述

---

## 用户确认断点（强制锁）

- 在 Phase 2.5 输出最终计划后，必须显示以下语句并停止，直到用户明确回复“执行”或批准：
  > “以上是修订后的执行计划，请确认是否符合预期。你可以要求调整步骤、优先级或范围。确认后我将开始执行。”
- 绝不允许在未收到确认的情况下进入 Phase 3。

## 反面案例：本次 /model 命令实现

| 阶段 | 应该做 | 实际做了 | 后果 |
|------|--------|---------|------|
| Comprehend | 追问“交互式”的具体定义 | 自行脑补弹窗 UI | 多写 100+ 行无用代码 |
| Orient | 读 CLI 入口，确认 `new TuiApp` | 跳过，直接改 `chat-ui.ts` | 前 3 个 commit 全部无效 |
| Decide | 最小文件清单 2 个 + 子任务拆分 | 无清单，边改边加 | 波及 6 个文件 |
| Act | 每步验证 | 连写 3 个 commit 然后修补 | 4 个 commit 中 3 个返工 |
| Verify | 追踪入口调用链 | 等用户反馈 bug 再排查 | 第二轮返工 |

## 正面示例：给 CLI 工具增加日志功能

### 复杂度判定
- 涉及日志模块 + 修改命令执行流 → 2 个功能点，判定为复杂，进入规划。

### Phase 0 验收标准
- 人视角：任何命令执行后都应有日志记录到文件，包含时间、命令、结果。
- 技术视角：运行 `tool --log test` 后，`/var/log/tool.log` 有新条目，`grep` 可验证。

### Phase 1 调用链定位
入口 `cli/index.ts` → `CommandFactory.create()` → `execute()` 方法 → 返回值处理。插入点：`execute()` 前后。

### Phase 2 文件清单 & 子任务
| ID | 目标 | 输入 | 输出 | 验收 | 风险 |
|----|------|------|------|------|------|
| 1 | 引入 winston 并创建基础 logger | 无 | `src/logger.ts` | 单测验证 logger 可正常创建并写入 | 日志库版本冲突，缓解：锁定版本 |
| 2 | 封装高阶函数包装 execute 方法 | `src/logger.ts`, `src/commands/*.ts` | 修改后的命令文件 | 运行任意命令，日志文件有新记录 | 可能中断异步执行流，缓解：先只包装同步路径 |

### Phase 2.5 自检表（示例，实际输出需完整）
发现缺失：初始计划未考虑日志级别控制 → 在子任务 2 追加配置项，标记 `[→ 已优化]`。

### Phase 3 执行
commit 1: 添加 logger 模块及单测  
commit 2: 重构命令执行层，引入日志包装  
commit 3: 增加配置开关及最终集成测试  

### Phase 4 验证
- 模拟运行 `tool --log test`，确认日志文件生成。
- grep 调用方，确认所有命令路径均已包装。

## 技巧
- **用 git 保护自己**：不确定的改动在新分支上做，坏了就 `git checkout -- .`
- **grep 是你的侦察兵**：`grep -rn "new TuiApp\|ChatUI" src/cli/`
- **“不改”比“改”更难**：每多加一个文件，就是多加一个未来的回归风险
